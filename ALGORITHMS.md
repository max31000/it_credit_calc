## 2. Базовые параметры ипотеки

### 2.1 Аннуитетный платёж

Аннуитетный платёж — фиксированная сумма, выплачиваемая каждый месяц на протяжении всего срока кредита. Включает проценты и часть основного долга таким образом, что к концу срока долг обнуляется.

**Формула:**

```
PMT = P × r / (1 − (1 + r)^(−n))
```

где:
- `P` — первоначальная сумма кредита
- `r = annualRate / 12 / 100` — месячная процентная ставка в долях
- `n` — срок кредита в месяцах

**Частный случай: r → 0**

При нулевой ставке формула вырождается (деление на ноль). Предел:

```
PMT = P / n
```

**Частный случай: P = 0**

Если первоначальный взнос покрывает полную стоимость квартиры (`downPayment ≥ apartmentPrice`), то `P = 0` и `PMT = 0`. См. граничный случай 6.

---

### 2.2 Разбивка платежа на проценты и тело долга

В месяце `t` (1-индексация: t = 1, 2, …, n):

```
D(0) = P                                        // начальный долг

interest_t   = D(t−1) × r                      // часть на проценты
principal_t  = PMT − interest_t                 // часть на тело долга
D(t)         = D(t−1) − principal_t             // остаток долга после t-го платежа
```

**Свойства:**
- С каждым месяцем `interest_t` убывает, `principal_t` растёт
- При правильном расчёте `D(n) = 0` (долг погашен точно в срок)
- Небольшая числовая погрешность допустима (< 1 ₽) из-за округления

---

### 2.3 Итоговая переплата

```
totalPayments    = PMT × n          // сумма всех аннуитетных платежей
totalInterest    = PMT × n − P      // переплата банку (только проценты)
```

**Интерпретация:** `totalInterest` — это цена пользования деньгами банка за весь срок. При досрочном погашении реальная переплата меньше.

## 3. Стратегия А — досрочное погашение с уменьшением платежа

### 3.1 Описание

Каждый месяц клиент платит аннуитетный платёж (изначально PMT) плюс направляет весь свободный остаток на **досрочное погашение тела долга**. При этом выбирается режим **«уменьшение платежа»** (не сокращение срока): после каждого досрочного погашения пересчитывается новый аннуитет на оставшийся срок от нового, меньшего остатка долга.

**Ключевые следствия:**
- С каждым месяцем аннуитет `PMT_A(t)` уменьшается (т.к. долг убывает быстрее, чем по базовому плану)
- По мере уменьшения платежа растёт сумма досрочного погашения (т.к. `freeMonthly` фиксировано, а обязательный платёж падает)
- Срок кредита формально не меняется, но долг может обнулиться досрочно
- Накопления стратегии А = 0 (все деньги идут в погашение)

**Важное уточнение о «свободных деньгах»:**  
`freeMonthly` — это сумма, которую клиент готов тратить **сверх** минимального аннуитетного платежа. То есть фактические расходы клиента в месяц:
- Стратегия А: `PMT_A(t) + prepay_t = freeMonthly + PMT_A(t)` → нет, см. схему ниже.

**Уточнение схемы расчёта:**  
Предполагается, что клиент располагает суммой `freeMonthly` для направления поверх минимального платежа. Если текущий аннуитет `PMT_A(t) < freeMonthly`, то:

```
prepay_t = freeMonthly − PMT_A(t)
```

Таким образом, **суммарный расход клиента в месяц = PMT_A(t) + prepay_t = freeMonthly** (постоянный).  
Если `PMT_A(t) ≥ freeMonthly` (долг велик, свободных денег не хватает) — досрочного погашения нет, `prepay_t = 0`.

---

### 3.2 Псевдокод стратегии А

```
ВХОД: P, r, n, freeMonthly

D_A[0] ← P
PMT_A[1] ← P × r / (1 − (1+r)^(−n))         // начальный аннуитет

totalInterest_A ← 0
totalPayments_A ← 0

ДЛЯ t ОТ 1 ДО n:

    ЕСЛИ D_A[t−1] ≤ 0:
        // долг уже погашен, все последующие платежи = 0
        D_A[t] ← 0
        ПРОДОЛЖИТЬ

    // 1. Рассчитываем процентную часть
    interest_t ← D_A[t−1] × r

    // 2. Аннуитетная часть (тело + проценты)
    //    PMT_A[t] уже пересчитан на предыдущем шаге
    principal_t ← PMT_A[t] − interest_t

    // 3. Определяем досрочное погашение
    ЕСЛИ freeMonthly > PMT_A[t]:
        prepay_t ← freeMonthly − PMT_A[t]
    ИНАЧЕ:
        prepay_t ← 0

    // 4. Обновляем долг
    D_A[t] ← D_A[t−1] − principal_t − prepay_t

    // 5. Защита от отрицательного долга
    ЕСЛИ D_A[t] < 0:
        prepay_t ← prepay_t + D_A[t]           // уменьшаем последний досрочный платёж
        D_A[t] ← 0

    // 6. Учёт статистики
    actualPayment_t ← PMT_A[t] + prepay_t
    totalInterest_A ← totalInterest_A + interest_t
    totalPayments_A ← totalPayments_A + actualPayment_t

    // 7. Пересчёт нового аннуитета для следующего месяца
    remainingMonths ← n − t
    ЕСЛИ D_A[t] > 0 И remainingMonths > 0:
        PMT_A[t+1] ← D_A[t] × r / (1 − (1+r)^(−remainingMonths))
    ИНАЧЕ ЕСЛИ D_A[t] > 0 И remainingMonths = 0:
        PMT_A[t+1] ← D_A[t]                    // последний платёж = весь остаток
    ИНАЧЕ:
        PMT_A[t+1] ← 0                          // долг погашен

КОНЕЦ ЦИКЛА

// Накопления стратегии А = 0
S_A[t] ← 0  для всех t
```

---

### 3.3 Метрики стратегии А

| Метрика | Формула |
|---|---|
| Остаток долга в момент t | `D_A(t)` |
| Суммарные уплаченные проценты | `totalInterest_A` (накапливается в цикле) |
| Накопления | `S_A(t) = 0` для всех t |
| Чистое состояние | `netWorth_A(t) = S_A(t) − D_A(t) = −D_A(t)` |
| Экономия на процентах vs базового плана | `totalInterest_base − totalInterest_A` |

**Интерпретация netWorth_A:**  
`netWorth_A(t) = −D_A(t)` — отрицательное число, пока долг не погашен. Это не означает «убыток»: клиент владеет квартирой, стоимость которой не учитывается в данной метрике. Метрика показывает чисто финансовое состояние без учёта стоимости актива.

## 4. Стратегия Б — минимальный платёж + инвестиции

### 4.1 Описание

Клиент платит только фиксированный аннуитет `PMT` (рассчитанный один раз в начале). Всю сумму сверх минимального платежа (`freeMonthly − PMT`) клиент ежемесячно инвестирует (депозит, фондовый рынок и т.д.) под ставку `depositRate`.

**Логика:** если депозитная ставка выше ипотечной, выгоднее держать дешёвый долг и копить под высокую ставку. Накопления со временем превысят долг, формируя положительное чистое состояние.

**Условие применимости стратегии Б:**  
`freeMonthly > PMT` — только тогда есть что инвестировать. При `freeMonthly ≤ PMT` инвестиций нет, обе стратегии идентичны.

---

### 4.2 Псевдокод стратегии Б

```
ВХОД: P, r, n, freeMonthly, depositRate

PMT ← P × r / (1 − (1+r)^(−n))               // фиксируется один раз навсегда
r_dep ← depositRate / 12 / 100

D_B[0] ← P
S_B[0] ← 0

totalInterest_B ← 0
totalInvested_B ← 0

ДЛЯ t ОТ 1 ДО n:

    ЕСЛИ D_B[t−1] ≤ 0:
        // долг погашен ранее (если такое произошло)
        D_B[t] ← 0
        S_B[t] ← S_B[t−1] × (1 + r_dep)       // накопления продолжают расти
        ПРОДОЛЖИТЬ

    // 1. Разбивка платежа
    interest_t  ← D_B[t−1] × r
    principal_t ← PMT − interest_t
    D_B[t]      ← D_B[t−1] − principal_t

    // 2. Расчёт инвестируемой суммы
    invest_monthly ← МАКС(0, freeMonthly − PMT)

    // 3. Обновление накоплений: сначала рост, потом пополнение
    S_B[t] ← S_B[t−1] × (1 + r_dep) + invest_monthly

    // 4. Статистика
    totalInterest_B  ← totalInterest_B + interest_t
    totalInvested_B  ← totalInvested_B + invest_monthly

КОНЕЦ ЦИКЛА
```

**Порядок применения депозитного роста:** в формуле `S(t) = S(t−1) × (1 + r_dep) + invest_monthly` предполагается, что вложение месяца `t` происходит в **конце** месяца (не успевает нарасти за текущий месяц). Это консервативная оценка.

---

### 4.3 Метрики стратегии Б

| Метрика | Формула |
|---|---|
| Остаток долга в момент t | `D_B(t)` |
| Накопления в момент t | `S_B(t)` |
| Чистое состояние | `netWorth_B(t) = S_B(t) − D_B(t)` |
| Суммарные уплаченные проценты | `totalInterest_B` |
| Итого инвестировано (без дохода) | `totalInvested_B` |

## 5. Сценарий слёта с льготной ипотеки

### 5.1 Описание и формулы

В момент `T_slip` (условно — государство отменяет льготную ставку, или банк пересматривает условия). Клиент вынужден перейти на рыночные условия.

**Рыночная ставка:**

```
r_market (% годовых) = keyRate − bankDiscount + 1.5
```

Слагаемое `+1.5%` — типичная банковская маржа сверх ключевой ставки.  
Пример: `keyRate = 16%`, `bankDiscount = 0.5%` → `r_market = 16 − 0.5 + 1.5 = 17%`

```
r_m = r_market / 12 / 100                    // месячная рыночная ставка в долях
```

**Оставшийся срок** после слёта:

```
remainAfterSlip = n − T_slip                 // месяцев до конца ипотеки
```

---

**Для стратегии Б при слёте:**

В момент `T_slip` клиент использует **все накопленные инвестиции** для досрочного погашения:

```
D_new_B = МАКС(0, D_B(T_slip) − S_B(T_slip))
S_new_B = МАКС(0, S_B(T_slip) − D_B(T_slip))
```

**Для стратегии А при слёте:**

```
D_new_A = D_A(T_slip)
S_new_A = 0                                  // накоплений нет
```

---

**Новые аннуитеты после слёта:**

Платёж при слёте **для стратегии Б после применения накоплений** (с досрочным погашением):

```
PMT_slip_prepay = D_new_B × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
```

Платёж при слёте **для стратегии Б без применения накоплений** (без досрочного):

```
PMT_slip_noprepay = D_B(T_slip) × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
```

Платёж при слёте **для стратегии А**:

```
PMT_slip_A = D_A(T_slip) × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
```

**Граничный случай `remainAfterSlip = 0`:** кредит уже погашен, платёж = 0.  
**Граничный случай `D_new = 0`:** досрочно погашен, платёж = 0.

---

### 5.2 Псевдокод slipAnalysis

`slipAnalysis` — массив, где для каждого гипотетического момента слёта `t` (от 0 до `horizonMonths`) рассчитываются оба вида платежа:

```
ВХОД: массивы D_B[], S_B[], r_market, n

r_m ← r_market / 12 / 100
slipAnalysis ← пустой массив

ДЛЯ t ОТ 0 ДО horizonMonths:

    remainAfterSlip ← n − t

    ЕСЛИ remainAfterSlip ≤ 0:
        paymentNoPrepay[t] ← 0
        paymentWithPrepay[t] ← 0
        ПРОДОЛЖИТЬ

    // Без применения накоплений
    ЕСЛИ D_B[t] > 0:
        paymentNoPrepay[t] ← D_B[t] × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
    ИНАЧЕ:
        paymentNoPrepay[t] ← 0

    // С применением накоплений
    D_after_prepay ← МАКС(0, D_B[t] − S_B[t])
    ЕСЛИ D_after_prepay > 0:
        paymentWithPrepay[t] ← D_after_prepay × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
    ИНАЧЕ:
        paymentWithPrepay[t] ← 0

    slipAnalysis[t] ← {
        month: t,
        D_B: D_B[t],
        S_B: S_B[t],
        D_after_prepay: D_after_prepay,
        paymentNoPrepay: paymentNoPrepay[t],
        paymentWithPrepay: paymentWithPrepay[t]
    }

КОНЕЦ ЦИКЛА

ВЕРНУТЬ slipAnalysis
```

---

### 5.3 Псевдокод slipScenario

`slipScenario` — детальный расчёт для одного конкретного момента слёта `T_slip`:

```
ВХОД: D_A[], D_B[], S_B[], T_slip, r_market, n, PMT, freeMonthly, r_dep

r_m ← r_market / 12 / 100
remainAfterSlip ← n − T_slip

// Состояния на момент слёта
D_A_slip ← D_A[T_slip]
D_B_slip ← D_B[T_slip]
S_B_slip ← S_B[T_slip]

// Применяем накопления Б к долгу
D_new_B ← МАКС(0, D_B_slip − S_B_slip)
S_new_B ← МАКС(0, S_B_slip − D_B_slip)

// Новые платежи для Б после слёта
ЕСЛИ remainAfterSlip > 0 И D_new_B > 0:
    PMT_slip_prepay ← D_new_B × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
ИНАЧЕ:
    PMT_slip_prepay ← 0

ЕСЛИ remainAfterSlip > 0 И D_B_slip > 0:
    PMT_slip_noprepay ← D_B_slip × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
ИНАЧЕ:
    PMT_slip_noprepay ← 0

ЕСЛИ remainAfterSlip > 0 И D_A_slip > 0:
    PMT_slip_A ← D_A_slip × r_m / (1 − (1 + r_m)^(−remainAfterSlip))
ИНАЧЕ:
    PMT_slip_A ← 0

ВЕРНУТЬ {
    D_A_slip, D_B_slip, S_B_slip,
    D_new_B, S_new_B,
    PMT_slip_prepay, PMT_slip_noprepay, PMT_slip_A
}
```

## 6. Точка финансовой безопасности

### 6.1 Определение

**Точка финансовой безопасности** — первый месяц `t*`, при котором:

```
paymentWithPrepay(t*) ≤ PMT
```

**Смысл:** если в момент `t*` произойдёт слёт с льготной ипотеки, и клиент направит все накопленные по стратегии Б деньги на досрочное погашение, то новый обязательный ежемесячный платёж по рыночной ставке не превысит первоначальный льготный аннуитет `PMT`.

Иными словами, с этого момента клиент **финансово защищён** от роста ставки: накопления «страхуют» от повышения платежа.

**Почему сравниваем с PMT (льготным), а не с текущим доходом:**  
Логика консервативная — клиент уже адаптировался жить, платя `PMT` (плюс инвестируя разницу). Если новый платёж не превышает `PMT`, его бюджет не пострадает.

---

### 6.2 Алгоритм поиска

```
ВХОД: slipAnalysis[], PMT, horizonMonths

safetyPoint ← null

ДЛЯ t ОТ 0 ДО horizonMonths:
    ЕСЛИ slipAnalysis[t].paymentWithPrepay ≤ PMT:
        safetyPoint ← t
        ПРЕРВАТЬ

ВЕРНУТЬ safetyPoint   // null если точка не найдена в горизонте
```

**Монотонность:** `paymentWithPrepay(t)` в общем случае убывает (долг уменьшается, накопления растут), поэтому первое вхождение условия обычно и является ответом, но алгоритм перебора корректен в любом случае.

**Если `safetyPoint = null`:** за горизонт анализа точка безопасности не достигнута. Это означает, что стратегия Б не даёт достаточной защиты в рамках анализируемого периода. Рекомендуется увеличить горизонт или пересмотреть параметры.

## 7. Налоговые вычеты

### 7.1 Прогрессивная шкала НДФЛ 2025

Ставки НДФЛ зависят от совокупного годового дохода (`annualIncome = salary × 12`):

| Диапазон годового дохода | Ставка |
|---|---|
| До 2 400 000 ₽/год (≤ 200 000 ₽/мес) | 13% |
| 2 400 000 — 5 000 000 ₽/год | 15% |
| 5 000 000 — 20 000 000 ₽/год | 18% |
| Свыше 20 000 000 ₽/год | 20% |

**Упрощение для данного калькулятора:**  
Вместо точного посегментного расчёта применяется **единая маргинальная ставка**, определяемая по полному годовому доходу. Это даёт небольшую погрешность в пользу завышения для доходов вблизи границ диапазонов, но упрощает вычисления.

```
annualIncome ← salary × 12

ЕСЛИ annualIncome ≤ 2_400_000:
    ndflRate ← 0.13
ИНАЧЕ ЕСЛИ annualIncome ≤ 5_000_000:
    ndflRate ← 0.15
ИНАЧЕ ЕСЛИ annualIncome ≤ 20_000_000:
    ndflRate ← 0.18
ИНАЧЕ:
    ndflRate ← 0.20
```

---

### 7.2 Имущественный вычет (за покупку)

Налогоплательщик вправе вернуть НДФЛ с суммы, потраченной на покупку жилья, но не более 2 000 000 ₽.

**Корректная формула:**

```
propertyDeductionBase ← МИН(2_000_000, apartmentPrice)
propertyDeductionTotal ← propertyDeductionBase × ndflRate
```

**Важно:** база вычисляется от **стоимости квартиры** (`apartmentPrice`), а не от суммы кредита (`loanAmount`). Это принципиальное отличие: при большом первоначальном взносе кредит может быть меньше 2 млн, но вычет всё равно считается от полной стоимости квартиры (до лимита 2 млн).

**Выплата по годам:**  
Вычет можно получить только в размере фактически уплаченного НДФЛ за год:

```
maxReturnPerYear ← salary × 12 × ndflRate     // максимум НДФЛ, уплаченного за год

remainingDeduction ← propertyDeductionTotal
yearlyPropertyReturn[] ← пустой массив

ДЛЯ каждого года:
    returnThisYear ← МИН(remainingDeduction, maxReturnPerYear)
    yearlyPropertyReturn[год] ← returnThisYear
    remainingDeduction ← remainingDeduction − returnThisYear
    ЕСЛИ remainingDeduction ≤ 0: ПРЕРВАТЬ
```

---

### 7.3 Вычет по процентам (ипотечный)

Налогоплательщик вправе вернуть НДФЛ с суммы фактически уплаченных процентов по ипотеке.

**Лимит базы:** 3 000 000 ₽ (суммарно за все годы).  
**Максимальный возврат:** `3_000_000 × ndflRate`

```
interestDeductionLimit ← 3_000_000
cumulativeInterestPaid ← 0
cumulativeInterestDeduction ← 0

ДЛЯ каждого года Y (от 1):
    interestThisYear ← сумма interest_t за все месяцы года Y

    // Применяем лимит
    deductibleThisYear ← МИН(interestThisYear, interestDeductionLimit − cumulativeInterestPaid)
    cumulativeInterestPaid ← cumulativeInterestPaid + deductibleThisYear

    interestReturnThisYear ← deductibleThisYear × ndflRate

    // Ограничено уплаченным НДФЛ за год (с учётом уже использованного на имущественный вычет)
    remainingNDFL_Y ← МАКС(0, maxReturnPerYear − yearlyPropertyReturn[Y])
    interestReturnThisYear ← МИН(interestReturnThisYear, remainingNDFL_Y)

    yearlyInterestReturn[Y] ← interestReturnThisYear
    cumulativeInterestDeduction ← cumulativeInterestDeduction + interestReturnThisYear

    ЕСЛИ cumulativeInterestPaid ≥ interestDeductionLimit: ПРЕРВАТЬ (лимит исчерпан)
```

---

### 7.4 Псевдокод добавления вычетов в накопления

Налоговые возвраты добавляются к накоплениям в конце соответствующего года. Для стратегии Б они инвестируются:

```
ДЛЯ каждого года Y:
    taxReturn_Y ← yearlyPropertyReturn[Y] + yearlyInterestReturn[Y]

    // Добавляем в конце последнего месяца года Y (месяц Y×12)
    t_end ← Y × 12
    S_B[t_end] ← S_B[t_end] + taxReturn_Y

    // Для стратегии А налоговый возврат идёт на досрочное погашение
    prepay_A[t_end] ← prepay_A[t_end] + taxReturn_Y
    // ... и пересчитывается аннуитет
```

### 9.2 Вычисление PMT

```
r = 6 / 12 / 100 = 0.005

PMT = 5_500_000 × 0.005 / (1 − (1.005)^(−240))

Шаг 1: числитель = 5_500_000 × 0.005 = 27_500

Шаг 2: (1.005)^240
  ln(1.005) ≈ 0.004987542
  240 × 0.004987542 ≈ 1.197010
  e^1.197010 ≈ 3.30984
  → (1.005)^240 ≈ 3.30984

Шаг 3: (1.005)^(−240) = 1 / 3.30984 ≈ 0.30213

Шаг 4: знаменатель = 1 − 0.30213 = 0.69787

Шаг 5: PMT = 27_500 / 0.69787 ≈ 39_404 ₽/мес
```

**PMT ≈ 39 404 ₽/мес**

### 9.3 Вычисление D_B(36) — остаток по стратегии Б

По стратегии Б платится только базовый аннуитет, долг убывает по обычному графику.

Остаток долга после t платежей по аннуитетному кредиту вычисляется точной формулой:

```
D(t) = P × (1+r)^t − PMT × ((1+r)^t − 1) / r
```

```
D_B(36):

(1.005)^36:
  36 × ln(1.005) = 36 × 0.004987542 ≈ 0.179551
  e^0.179551 ≈ 1.19668
  → (1.005)^36 ≈ 1.19668

D_B(36) = 5_500_000 × 1.19668 − 39_404 × (1.19668 − 1) / 0.005
         = 5_500_000 × 1.19668 − 39_404 × 0.19668 / 0.005
         = 6_581_740 − 39_404 × 39.336
         = 6_581_740 − 1_549_965
         ≈ 5_031_775 ₽
```

**D_B(36) ≈ 5 031 775 ₽**

### 9.4 Вычисление S_B(36) — накопления по стратегии Б

```
invest_monthly = freeMonthly − PMT = 100_000 − 39_404 = 60_596 ₽/мес

r_dep = 16 / 12 / 100 ≈ 0.013333

Формула будущей стоимости аннуитета постнумерандо (взносы в конце месяца):
S_B(36) = invest_monthly × ((1 + r_dep)^36 − 1) / r_dep

(1.013333)^36:
  36 × ln(1.013333) = 36 × 0.013245 ≈ 0.476820
  e^0.476820 ≈ 1.61054
  → (1.013333)^36 ≈ 1.61054

S_B(36) = 60_596 × (1.61054 − 1) / 0.013333
         = 60_596 × 0.61054 / 0.013333
         = 60_596 × 45.791
         = 2_775_100 ₽
```

**S_B(36) ≈ 2 775 100 ₽**

### 9.5 Вычисление D_A(36) — остаток по стратегии А

По стратегии А каждый месяц направляется `freeMonthly = 100 000 ₽` суммарно.  
Из них `PMT_A(t)` — аннуитет (убывает), `prepay_t = 100_000 − PMT_A(t)` — досрочное.

На первом месяце: `PMT_A(1) ≈ 39 404`, `prepay_1 = 60 596 ₽`.

Для точного расчёта D_A(36) используем итерационный алгоритм. Ниже приведён **приближённый расчёт** через упрощённую модель:

За 36 месяцев суммарно направлено на погашение долга:
```
totalDirected = freeMonthly × 36 = 100_000 × 36 = 3_600_000 ₽
```

Из этих 3 600 000 ₽ часть ушла на проценты. Примерная оценка уплаченных процентов за 36 месяцев при быстро убывающем долге — сложнее базового случая, поэтому приводим границы:

- **Нижняя граница** (как если бы долг сразу снизился): проценты за 36 мес при долге ~3.5M ≈ 3.5M × 0.005 × 36 ≈ 630 000 ₽
- **Верхняя граница** (как в стратегии Б, 36 мес по базовому графику): ~469 000 ₽ (стандартные начисленные проценты)

Точный расчёт итерацией:

```
// Приближённая итерация (упрощённо, без пересчёта аннуитета каждый шаг)
D ← 5_500_000
totalInterest_36 ← 0

// Месяц 1:

PMT_A = 39_404 (начальный)
principal = 39_404 − 27_500 = 11_904
prepay = 100_000 − 39_404 = 60_596
D ← 5_500_000 − 11_904 − 60_596 = 5_427_500

// В реальности каждый следующий месяц PMT_A пересчитывается.
// После первого досрочного погашения:
// PMT_A(2) = 5_427_500 × 0.005 / (1 − 1.005^(−239)) ≈ 38_997 ₽
// prepay(2) = 100_000 − 38_997 = 61_003 ₽ — чуть больше

// После 36 месяцев суммарные проценты ≈ 420_000–450_000 ₽
// Тело погашено ≈ 3_600_000 − 435_000 = 3_165_000 ₽

D_A(36) ≈ 5_500_000 − 3_165_000 ≈ 2_335_000 ₽
```

**D_A(36) ≈ 2 335 000 ₽** (оценка; точное значение требует полной итерации)

> Для точного значения необходима компьютерная итерация. Оценка консервативна: реальное значение может быть в диапазоне **2 200 000 — 2 400 000 ₽**.

## 10. Граничные случаи

### 10.1 Досрочное погашение полного долга (стратегия А)

**Условие:** `D_A(t) ≤ 0` в некоторый месяц `t* < n`

**Поведение:**
```
ЕСЛИ D_A[t] ≤ 0:
    // Корректируем последний платёж, чтобы не переплатить
    actualPrepay ← actualPrepay + D_A[t]    // уменьшаем на "лишнее"
    D_A[t] ← 0

// Для всех t > t*:
    D_A[t] ← 0
    PMT_A[t] ← 0
    prepay_t ← 0
    interest_t ← 0
```

Накопления стратегии А остаются нулевыми. После погашения долга у клиента освобождаются `freeMonthly` — но это выходит за рамки текущей модели.

---

### 10.2 Слёт в первый месяц (T_slip = 0 или 1)

**Состояние:** `D_B(0) = P`, `S_B(0) = 0`

```
D_new_B = МАКС(0, P − 0) = P
PMT_slip_prepay = P × r_m / (1 − (1 + r_m)^(−n))
```

Платёж при слёте в первый месяц — это просто аннуитет по рыночной ставке на полную сумму кредита. Ожидаемо высокий.

---

### 10.3 Слёт в последний месяц горизонта (T_slip = horizonMonths)

**Состояние:** `D_B(horizonMonths)` — небольшой остаток (если горизонт < n) или 0; `S_B(horizonMonths)` — большое накопление.

```
// Если S_B >> D_B:
D_new_B = 0    // накопления с запасом покрывают долг
S_new_B = S_B − D_B > 0

PMT_slip_prepay = 0   // долг погашен, платежей нет
```

Это лучший возможный исход для стратегии Б.

---

### 10.4 freeMonthly < PMT

**Условие:** клиент не может платить даже минимальный аннуитет с запасом.

```
invest_monthly = МАКС(0, freeMonthly − PMT) = 0   // нет инвестиций
prepay_t = МАКС(0, freeMonthly − PMT_A[t]) = 0   // нет досрочного

// Обе стратегии идентичны базовому аннуитетному плану:
D_A(t) = D_B(t) = D_base(t)
S_A(t) = S_B(t) = 0
netWorth_A(t) = netWorth_B(t) = −D_base(t)
```

**Следствие:** победитель не определён (ничья), калькулятор должен информировать пользователя.

---

### 10.5 depositRate = 0

```
r_dep = 0

S_B(t) = S_B(t−1) × 1 + invest_monthly = S_B(t−1) + invest_monthly
// Накопления растут линейно, без процентного дохода:
S_B(t) = invest_monthly × t
```

В этом случае стратегия А с высокой вероятностью выгоднее (досрочное погашение уменьшает проценты, а депозит не приносит дохода).

---

### 10.6 downPayment ≥ apartmentPrice (кредит не нужен)

```
loanAmount = МАКС(0, apartmentPrice − downPayment) = 0
P = 0
PMT = 0

D(t) = 0 для всех t
totalInterest = 0

// Стратегия Б: инвестируется весь freeMonthly
invest_monthly = freeMonthly   // (т.к. PMT = 0)

// Стратегия А: нет долга, нет досрочного погашения
// Накоплений нет, netWorth_A = 0
```

Этот случай нужно обработать отдельно: нет смысла сравнивать стратегии, рекомендовать инвестирование.

## 11. Расширения

> Раздел помечен как **расширение** — не входит в базовую функциональность, но рекомендован для полноты анализа.

---

### 11.1 Метрика «Экономия от досрочного погашения»

Показывает, сколько процентов сэкономила стратегия А по сравнению с базовым аннуитетным планом (без досрочного погашения):

```
savingsFromPrepayment = totalInterest_base − totalInterest_A
```

где `totalInterest_base = PMT × n − P` — переплата по базовому плану без досрочки.

**Интерпретация:** это «стоимость льготного долга», которую стратегия А избегает, ускоряя погашение.

---

### 11.2 Метрика «Стоимость страховки досрочного погашения» (opportunity cost)

Альтернативная стоимость: сколько стратегия А «недозаработала» по сравнению со стратегией Б за счёт направления денег в долг вместо инвестиций:

```
opportunityCost = S_B(horizonMonths) − S_A(horizonMonths)
               = S_B(horizonMonths) − 0
               = S_B(horizonMonths)
```

**Интерпретация:** если стратегия Б накопила 5 млн на счету, а стратегия А — 0, то цена «страховки» (досрочного погашения как защиты от ставки) составляет 5 млн ₽.

Более корректная метрика с учётом экономии на процентах:

```
netOpportunityCost = S_B(horizonMonths) − savingsFromPrepayment
```

Если `netOpportunityCost > 0` — стратегия Б выгоднее. Если `< 0` — стратегия А выгоднее.

---

### 11.3 График точки безубыточности (Break-even Point)

Точка безубыточности — месяц `t_be`, при котором:

```
netWorth_B(t_be) = netWorth_A(t_be)
S_B(t_be) − D_B(t_be) = −D_A(t_be)
S_B(t_be) = D_B(t_be) − D_A(t_be)
```

**Алгоритм поиска:**

```
breakEvenPoint ← null

ДЛЯ t ОТ 1 ДО horizonMonths:
    ЕСЛИ netWorth_B(t) ≥ netWorth_A(t):
        breakEvenPoint ← t
        ПРЕРВАТЬ

ВЕРНУТЬ breakEvenPoint
```

**Характер кривых:**  
- `netWorth_A(t) = −D_A(t)` — монотонно возрастает (долг убывает), начинает с `−P`
- `netWorth_B(t) = S_B(t) − D_B(t)` — начинает убывать (долг > накоплений), затем растёт по мере накопления инвестиций

В типичном случае (ставка депозита >> ипотечная ставка) кривая Б пересекает кривую А снизу вверх в некоторый момент `t_be`. После этого Б устойчиво лидирует.

**График** (псевдо-данные для визуализации):

```
t=0:    A: −5_500_000    B: −5_500_000     (равны)
t=12:   A: ~−4_700_000   B: ~−5_180_000    (A лидирует)
t=36:   A: ~−2_335_000   B: ~−2_256_675    (A лидирует, разрыв сужается)
t=t_be: A = B                               (точка безубыточности)
t>t_be: A < B                               (B лидирует)
```

### З-1. Ошибка в исходном промпте: база имущественного вычета

**В исходном задании написано:**  
> База = min(2_000_000, loanAmount)

**Корректно:**  
> База = min(2_000_000, apartmentPrice)

Налоговый кодекс РФ (ст. 220) устанавливает вычет на расходы по **приобретению** жилья, а не по сумме кредита. Клиент мог купить квартиру за 10 млн, взяв ипотеку лишь на 1 млн — но вычет всё равно составит min(2_000_000, 10_000_000) × ndflRate. В документе используется корректная формула.

### З-2. Определение «свободных денег» (freeMonthly)

В промпте `freeMonthly` трактуется неоднозначно — то как «сверх минимального платежа», то как «весь свободный остаток». В данном документе принята следующая семантика:

**`freeMonthly`** — сумма, которую клиент готов суммарно тратить в месяц на ипотечные цели (минимальный платёж + досрочное / инвестиции). То есть суммарный ежемесячный расход клиента фиксирован и равен `freeMonthly` в обеих стратегиях.

Следствие: `invest_monthly = freeMonthly − PMT`, `prepay_t = freeMonthly − PMT_A(t)`.

---

### З-3. Прогрессивная шкала НДФЛ — упрощение

В документе применяется единая маргинальная ставка для всей суммы вычета. Точный расчёт требовал бы вычисления налога сначала с дохода, затем с дохода за вычетом базы — и возврата разницы. При доходах вблизи пороговых значений (200 000, ~417 000, ~1 667 000 ₽/мес) погрешность может достигать 2–5% от суммы вычета. Для калькулятора это допустимо, но должно быть отражено в UI как «приближённый расчёт».

---

### З-4. Тестовое значение D_A(36) является оценочным

Точный расчёт `D_A(36)` требует полной итерации 36 шагов с пересчётом аннуитета на каждом шаге (что в псевдокоде опущено для краткости). Приведённое значение ~2 335 000 ₽ — результат приближённого расчёта. В реализации необходимо использовать полный итерационный алгоритм.

---

### З-5. Момент досрочного погашения внутри месяца

Алгоритм предполагает, что досрочное погашение происходит **в конце месяца**, после начисления процентов. Если банк допускает досрочное погашение в начале месяца (до начисления процентов), формулы разбивки платежа меняются. Текущая модель — консервативная (проценты начисляются на полный остаток месяца).

---

### З-6. Налоговые вычеты — год начала получения

В модели предполагается, что имущественный вычет начинает выплачиваться с первого года. На практике первый возврат приходит по итогам декларации за первый год владения, то есть в начале второго года. Это смещение должно быть учтено в детальной реализации.

---

### З-7. Стоимость квартиры не учитывается в netWorth

В обеих стратегиях клиент владеет квартирой. Её рыночная стоимость не учитывается в `netWorth`. Это допустимо, поскольку обе стратегии предполагают одинаковый актив (квартира одна и та же), и для сравнения стратегий между собой её стоимость сокращается. Если цель — расчёт абсолютного состояния клиента, необходимо добавить `apartmentValue(t)` в обе формулы.

---

*Конец документа. Версия 1.0. Все формулы основаны на стандартных правилах банковских аннуитетных расчётов и налоговом законодательстве РФ (2025).*
